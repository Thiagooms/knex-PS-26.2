---

> Depois do banco de dados, criamos o pacote que reúne o que é
comum a todos os outros módulos, mas que não é responsabilidade de
nenhum deles especificamente.

### Erros com significado

Em vez de espalhar `throw new Error("deu ruim")` pelo código, foram criadas 
classes de erro com nome e propósito claro, por exemplo, um erro
específico pra "esse produto não tem estoque suficiente" e outro pra
"esse produto já foi vendido e não pode ser excluído". Cada erro já sabe
qual código de status HTTP deve retornar pro cliente da API.

### Um formato de erro único

Toda vez que a API responde com um erro, seja porque os dados enviados
estão errados, seja porque uma regra de negócio foi violada, a resposta
segue sempre o mesmo formato. Isso facilita muito a vida de quem for
consumir essa API depois, não importa o que deu errado, o formato da
resposta é sempre previsível.

### Validação de entrada

Também vive aqui uma peça que verifica se os dados que chegam numa
requisição têm o formato esperado, antes desses dados chegarem na lógica
de negócio de cada rota. Ela é genérica o suficiente pra ser usada por
qualquer rota do sistema, sem repetir código.

Nada disso depende de banco de dados, nem de nenhuma regra específica de
autenticação, produto ou pedido, é por isso que esse pacote existe
separado dos outros.
